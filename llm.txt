# PgCache Documentation

PgCache is a transparent caching proxy for PostgreSQL that sits between your application and database. It automatically caches SELECT query results and uses Change Data Capture (CDC) via logical replication to keep the cache synchronized with the origin.

GitHub: https://github.com/tempest98/pgcache

---

## Getting Started

### Prerequisites

- PostgreSQL 16+ as origin database
- Logical replication enabled on origin (`wal_level = logical`)
- Docker for running PgCache

### Configure Origin Database

Enable logical replication in `postgresql.conf`:

```
wal_level = logical
max_replication_slots = 10
max_wal_senders = 10
```

Grant replication permissions to user:

```
ALTER ROLE pgcache_user REPLICATION;
```

### Install with Docker

```bash
docker run -d -p 5432:5432 -p 9090:9090 pgcache/pgcache \
  --upstream postgres://user:password@your-db-host:5432/myapp
```

### Docker Compose Example

```yaml
services:
  pgcache:
    image: pgcache/pgcache
    ports:
      - "5432:5432"
      - "9090:9090"
    environment:
      UPSTREAM_URL: postgres://user:password@db:5432/myapp
      NUM_WORKERS: 4
      PUBLICATION: pgcache_pub
      SLOT: pgcache_slot
```

### Connect Your Application

Update your connection string to point at PgCache:

```
# Before
DATABASE_URL=postgres://user:password@db-host:5432/myapp

# After
DATABASE_URL=postgres://user:password@pgcache-host:5432/myapp
```

### Verify It Works

```bash
curl http://localhost:9090/metrics | grep pgcache_queries
```

Look for `pgcache_queries_cache_hit` and `pgcache_queries_cache_miss` counters.

## Architecture

```
┌──────────┐    queries     ┌──────────┐    uncacheable    ┌──────────┐
│          │ ─────────────▶ │          │ ────────────────▶ │          │
│   App    │                │ pgcache  │                   │PostgreSQL│
│          │ ◀───────────── │          │ ◀──────────────── │ (origin) │
└──────────┘   responses    └────┬──┬──┘    responses      └─────┬────┘
                                 │  │                            │
                           cache │  │    CDC stream               │
                           read/ │  │   (logical replication)    │
                           write │  └────────────────────────────┘
                                 ▼
                            ┌──────────┐
                            │  Cache   │
                            │   DB     │
                            └──────────┘
```

1. Application sends queries to PgCache as if it were PostgreSQL
2. PgCache analyzes each query to determine if it's cacheable
3. Cacheable queries are served from cache database on cache hit
4. Cache misses and non-cacheable queries are forwarded to origin
5. CDC stream continuously processes changes from origin

## Configuration

### Configuration Methods (Priority: CLI > TOML > Environment)

| Method | Usage | Best For |
|--------|-------|----------|
| Environment variables | `UPSTREAM_URL=postgres://...` | Docker deployments (recommended) |
| CLI arguments | Passed via Docker command | Overriding specific values |
| TOML file | Mounted into container | Version-controlled config |

### TOML Configuration Reference

#### [origin] — Origin Database Connection

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `host` | string | yes | — | Origin database hostname |
| `port` | integer | yes | — | Origin database port |
| `user` | string | yes | — | Database user |
| `password` | string | no | — | Database password |
| `database` | string | yes | — | Database name |
| `ssl_mode` | string | no | `disable` | TLS mode: `disable`, `require`, or `verify-full` |

#### [replication] — Replication Connection (Optional)

Override connection settings for CDC. Useful when app connects through PgBouncer but CDC needs direct PostgreSQL connection.

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `host` | string | origin host | Replication host |
| `port` | integer | origin port | Replication port |

| `password` | string | origin password | Replication password |
| `database` | string | origin database | Replication database |
| `ssl_mode` | string | origin ssl_mode | TLS mode |

| Field | Type | Required | Description |
|-------|------|----------|-------------|

| `slot_name` | string | yes | Name of replication slot PgCache creates on origin |

| Field | Type | Required | Description |

|-------|------|---------|-------------|

### Full Example Configuration

```toml
num_workers = 4
cache_size = 1073741824  # 1 GB
log_level = "info"

[origin]
host = "db.example.com"
port = 5432
```

## What Gets Cached

### Cacheable Queries

- Single-table SELECT statements
- INNER JOIN, LEFT JOIN, RIGHT JOIN with equality join conditions
- WHERE clauses: `=`, `!=`, `<`, `<=`, `>`, `>=`
- Boolean operators: AND, OR
- IN and NOT IN expressions
- IS NULL and IS NOT NULL checks
- NOT expressions
- GROUP BY and HAVING clauses
- ORDER BY clauses
- Aggregate functions: COUNT, SUM, AVG, etc.
- Window functions: ROW_NUMBER, RANK, etc.
- CASE expressions
- Arithmetic expressions
- Uncorrelated subqueries in SELECT, FROM, WHERE
- Correlated subqueries (EXISTS, NOT EXISTS, IN, NOT IN, scalar) - decorrelated into equivalent JOINs
- CTEs including MATERIALIZED/NOT MATERIALIZED
- Set operations: UNION, UNION ALL, INTERSECT, EXCEPT
- Functions in SELECT list
- Immutable functions in WHERE/JOIN: lower(), upper(), abs(), etc.
- LIMIT/OFFSET (queries differing only in LIMIT share cache entry)
- Pattern matching: LIKE, NOT LIKE, ILIKE, NOT ILIKE
- Range expressions: BETWEEN, NOT BETWEEN, BETWEEN SYMMETRIC
- Array comparisons: `= ANY(...)`, `op ALL(...)`
- Mixed wildcards and columns: `SELECT *, id, name FROM ...`
- EXISTS/NOT EXISTS subqueries
- IS [NOT] TRUE/FALSE/UNKNOWN
- Extended query protocol (parameterized queries)

### NOT Cached (Forwarded to Origin)

- Non-SELECT statements: INSERT, UPDATE, DELETE, DDL
- FULL JOIN and CROSS JOIN
- LATERAL subqueries
- Non-equality correlated predicates (`<`, `>`, OR-connected)
- RECURSIVE CTEs
- Non-immutable functions outside SELECT list
- Locking clauses: FOR UPDATE, FOR SHARE, etc.
- Views

## Cache Maintenance

### CDC (Change Data Capture)

1. PgCache creates publication and replication slot on origin
2. Publication is dynamically managed (only tables with cached queries)
3. Changes (INSERT, UPDATE, DELETE, TRUNCATE) streamed via logical replication
4. Affected cache entries updated directly or invalidated

### Table Allowlist

Restrict caching to specific tables:

```toml
allowed_tables = ["users", "orders", "products"]
```

### Pinned Queries

Pre-populate and protect queries from eviction:

```toml
pinned_queries = [
    "SELECT * FROM settings",
    "SELECT * FROM categories",
]
pinned_tables = ["settings", "categories"]
```

Single-table pinned queries are updated in place via CDC, never invalidated.

### Cache Admission

Default CLOCK policy requires query seen `admission_threshold` times (default 2) before caching. Prevents one-off queries consuming cache space. Fast-readmission after CDC invalidation.

### Predicate Subsumption

When query result set is covered by existing cached query. Example: if `SELECT * FROM users` is cached, `SELECT * FROM users WHERE id = 1` can be served from cached superset.

### Cache Size Management

When `cache_size` is set, CLOCK (second-chance) eviction policy applies. FIFO also available.

---

## Consistency Guarantees

PgCache provides **eventual consistency**:

- Changes streamed via CDC with minimal delay
- Replication lag exposed as metrics: `pgcache.cdc.lag_seconds`, `pgcache.cdc.lag_bytes`
- Cache fully reset on every PgCache startup
- Uncacheable queries always go directly to origin

Typical lag between origin write and cache invalidation: sub-second.

## AWS Marketplace

Pre-built AMI on Amazon Linux 2023 (ARM64/Graviton) with embedded PostgreSQL 18.

### Quick Start

1. Store database URL in SSM:
```bash
aws ssm put-parameter --name "/pgcache/prod/upstream-url" \
  --type SecureString \
  --value "postgres://user:password@host:5432/dbname?sslmode=require"
```

2. Create IAM role with SSM read access

3. Create security group (ports 5432, 9090)

4. Launch instance with bootstrap:
```bash
/opt/pgcache/bootstrap.sh --ssm-prefix /pgcache/prod
```

### Bootstrap Options

| Flag | Default | Description |
|------|---------|-------------|
| `--ssm-prefix` | (required) | SSM Parameter Store path prefix |
| `--workers` | half vCPUs | Proxy worker threads |
| `--cache-size` | unlimited | Max cache size in bytes |
| `--cache-policy` | `clock` | Eviction policy |
| `--admission-threshold` | `2` | Queries seen before caching |
| `--allowed-tables` | all | Comma-separated tables |
| `--pinned-tables` | none | Comma-separated tables to pin |
| `--pinned-queries` | none | Semicolon-separated queries |
| `--cdc-suffix` | EC2 instance ID | Suffix for publication/slot names |

| Type | vCPUs | Memory | Workers | Shared Buffers |

|------|-------------|

## Compatibility

### PostgreSQL Version Support

| Version | Status |
|---------|--------|
| 18 | Supported (primary target) |
| 17 | Supported (search_path changes not supported) |
| 16 | Supported (search_path changes not supported) |
| 15 and below | Not supported |

Origin must have `wal_level = logical` enabled.

### Client Library Compatibility

Any PostgreSQL client library works:

| Language | Libraries |
|----------|-----------|
| Python | psycopg2, psycopg3, asyncpg, SQLAlchemy |
| JavaScript/TypeScript | node-postgres, Prisma, Drizzle, Knex |
| Java | JDBC, HikariCP, jOOQ |
| Go | pgx, database/sql with pq driver |
| Rust | tokio-postgres, sqlx, diesel |
| Ruby | pg gem, ActiveRecord |
| .NET | Npgsql, Entity Framework |

### TLS Support

| Connection | Configuration |
|------------|---------------|
| Client to PgCache | `tls_cert` and `tls_key` (PEM) |
| PgCache to origin | `ssl_mode = "require"` or `"verify-full"` |
| PgCache to replication | `ssl_mode = "require"` or `"verify-full"` |

Note: Set `channel_binding=disable` in client connection string.

### Connection Pooler Compatibility

Works with PgBouncer:
- App → PgCache (caching proxy)
- PgCache → PgBouncer (for regular queries via `[origin]`)
- PgCache → PostgreSQL directly (for CDC via `[replication]`)

### Known Limitations

- Cache reset on startup (ensures clean state)
- Single-node only (no distributed cache)
- COPY protocol not proxied
- LISTEN/NOTIFY not proxied
- WAL can accumulate if PgCache down extended period

```bash
docker run -d -p 5432:5432 -p 9090:9090 \
  -v ./pgcache.toml:/etc/pgcache/config.toml:ro \
  -e CONFIG_FILE=/etc/pgcache/config.toml \
  pgcache/pgcache
```

With table allowlist and pinned tables:
```bash
docker run -d -p 5432:5432 \
  -e UPSTREAM_URL=postgres://user@db:5432/myapp \
  -e ALLOWED_TABLES=users,orders,products \
  -e PINNED_TABLES=users,products \
  pgcache/pgcache
```

With pinned queries:
```bash
docker run -d -p 5432:5432 pgcache/pgcache \
  --upstream postgres://user@db:5432/myapp \
  --pinned-queries "SELECT * FROM config;SELECT * FROM lookup"
```
